# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

This is a bilingual (zh/en) WLAN technical blog built with Obsidian + Hugo + GitHub Pages. The author is a WLAN baseband algorithm engineer. Writing happens in Obsidian; the Hugo site is a read-only build artifact generated by scripts.

**Key principle**: `obsidian-vault/posts/` is the single source of truth for content. Everything under `hugo/content.*/` is script-generated and should never be edited directly.

## Common commands

```bash
# Local Hugo dev server
cd hugo && hugo server --buildDrafts

# Production build
cd hugo && hugo --minify --cleanDestinationDir

# Sync Obsidian → Hugo (run before previewing)
bash scripts/sync.sh --force
```

### Claude Code skills (preferred)

| Command | What it does |
|---------|-------------|
| `/translate` or "translate" | Incremental translation: sync → detect changed posts → translate to EN |
| `/publish` or "publish" | Full pipeline: translate → build → commit → push |
| `publish --no-push` | Same but skip git push (local preview) |
| `/pre-push-check` | Run before pushing: validates CI config, navigation, math, build |

### Shell scripts (fallback)

```bash
bash scripts/translate.sh              # sync + detect (tells you to use /translate)
uv run python scripts/translate.py --status    # show translation coverage
bash scripts/publish.sh --no-push      # sync → build only (no push)
```

Hugo, uv, and the hugo-book theme are managed cross-platform:
- **First run on a new machine**: `bash scripts/setup.sh` installs missing tools (hugo, uv) and pulls the theme submodule
- **Clone**: `git clone --recurse-submodules <repo-url>` — the theme is a git submodule
- Python scripts run via `uv` (`uv run python ...`)

## Architecture

### Content flow

```
obsidian-vault/posts/          ← human edits here (Chinese)
    │
    ▼  sync.sh (copies non-draft posts)
hugo/content.zh/posts/         ← Chinese source for Hugo
    │
    ▼  /translate skill (Claude Code → DeepSeek API)
hugo/content.en/posts/         ← English translation for Hugo
    │
    ▼  hugo build
hugo/public/                   ← static site (gitignored)
```

`sync.sh` also copies `obsidian-vault/assets/` → `hugo/static/assets/` and `code/` → `hugo/static/code/`, making code files accessible on the website at `/code/matlab/...` etc.

### Multilingual setup

Hugo uses `contentDir` per language (`content.zh`, `content.en`) to match the hugo-book theme convention. Without `contentDir`, the theme's `BookSection` lookup fails. This is the same approach used in the theme's own exampleSite.

### Translation pipeline

Translation is a **two-layer system**:

1. **Detection + cache** (`scripts/translate.py`): Scans `hugo/content.zh/posts/`, computes MD5 hashes, and compares against `scripts/.translate_cache/` to find files that are new or modified since last translation.

2. **Translation** (`/translate` skill at `.claude/skills/translate.md`): Invoked by saying "translate" in Claude Code. Runs sync → detect → translates each changed file using Claude's LLM (backed by DeepSeek per the user's Claude CLI config) → marks cache as done. The skill includes the full WLAN terminology glossary (140+ terms) and rules for preserving code blocks, math blocks, and Markdown formatting.

Cache is MD5-based so only genuinely changed files get re-translated.

### Hugo theme

Uses [hugo-book](https://github.com/alex-shpak/hugo-book), a documentation-oriented theme. Key config: `BookSection: posts` renders the `posts/` section as sidebar navigation. `BookTranslatedOnly: true` hides language options when no translation exists. KaTeX is enabled via Goldmark passthrough extensions for `$$...$$` and `$...$` delimiters.

### CI/CD

`.github/workflows/deploy.yml` triggers on push to `main`. It checks out with submodules, runs `hugo --minify` in `./hugo`, and deploys to GitHub Pages using official actions. The translated `.en/` content is committed to the repo (translation happens locally, not in CI), so CI only needs Hugo.

## Content authoring rules

Every post in `obsidian-vault/posts/` must have YAML frontmatter with these fields:

```yaml
---
title: "Post Title"
date: 2026-05-30
lastmod: 2026-05-30
tags: [WLAN, 802.11, PHY]
categories: [dive-into-wlan, phy-layer]
slug: "url-friendly-slug"
type: docs
BookToC: true
draft: true      # set to false when ready to publish
---
```

`sync.sh` skips drafts (frontmatter `draft: true`) unless `--force` is passed. Use the Obsidian template at `obsidian-vault/templates/post.md` when creating new posts.

## Files that must never be committed

- `scripts/.translate_cache/` — local translation cache
- `hugo/public/` and `hugo/resources/` — build output
